# 第 21 课：DataStream 如何序列化字符串

## 一、本课目标

完成本课后，你应该能够：

1. 理解字符串序列化和基本数值类型序列化的差别
2. 看懂 `DataStream` 中字符串写入和读取的基本流程
3. 理解“变长数据必须额外记录长度”这一核心规则
4. 明白字符串在字节流里通常由哪些部分组成
5. 为后续学习容器序列化建立统一思路

---

## 二、为什么字符串比整数更值得单独拿出来讲

前面几课你已经逐步建立了一个认知：

- `bool`、`char`、`int32_t`、`uint64_t`、`float`、`double` 这些类型的一个共同特点是：
  **长度基本固定**。

例如：
- `int32_t` 就是 4 字节
- `uint64_t` 就是 8 字节
- `double` 就是 8 字节

所以只要你知道：
- 当前类型是什么
- 从当前位置开始读取多少字节

就能把值取出来。

但字符串不一样。

字符串的核心特点是：

> **它是变长数据。**

`"a"` 和 `"middleware"` 的长度显然不同。

这就意味着：

- 不能只靠类型标识来恢复字符串
- 还必须知道这个字符串到底占多少字节

所以字符串序列化会比固定长度数值类型多一个关键步骤：

> **记录长度。**

这正是本课的重点。

## 四、先看字符串类型标识

在 `DataType` 枚举中，字符串对应的类型是：

- [../serialize/data_stream.h#L40-L46](../serialize/data_stream.h#L40-L46)

其中包含：

- `STRING`

这说明 `DataStream` 处理字符串时，仍然遵守前面几课已经建立的统一模式：

> **先写类型标识，再写具体内容。**

也就是说，字符串不是裸写进去的，而是带着“我是字符串”这个标签进入流。

这会让反序列化阶段更容易判断当前数据该如何解释。

## 五、`write(const char*)` 是怎么工作的

实现见：
- [../serialize/data_stream.cpp#L272-L279](../serialize/data_stream.cpp#L272-L279)

代码逻辑可以拆成 3 步：

### 第一步：写入字符串类型标识
```cpp
char type = DataType::STRING;
write((char *)&type, sizeof(char));
```

这一步告诉数据流：

- 接下来要写的是字符串

---

### 第二步：计算字符串长度并写入长度
```cpp
int len = strlen(value);
write(len);
```

这是本课最关键的一步。

为什么必须写长度？

因为字符串是变长数据。
如果不把长度写进去，读取方只知道：
- 这里是字符串

但并不知道：
- 要读几个字节才算这个字符串结束

所以长度字段实际上是字符串内容的“边界说明”。

---

### 第三步：写入字符串内容本身
```cpp
write(value, len);
```

这一步才是真正把字符数据拷贝到缓冲区。

注意这里写的是：
- `len` 个字符内容

而不是额外再写一个 `\0` 结束符。

这说明该实现对字符串边界的管理，不依赖 C 风格字符串的结尾约定，而依赖：

> **显式长度。**

这是一种更适合序列化格式的做法。

---

## 六、`write(const string&)` 的逻辑几乎一样

实现见：
- [../serialize/data_stream.cpp#L281-L288](../serialize/data_stream.cpp#L281-L288)

主要流程还是三步：

1. 写 `STRING` 类型标识
2. 写长度 `len`
3. 写实际字符数据 `value.data()`

也就是说，不管是 `const char*` 还是 `std::string`，本质编码规则都一样：

> **STRING + 长度 + 内容**

你一定要把这个模式记住。

因为后面你看容器序列化时，会发现它们其实也是类似的思路：

> **类型 + 元信息（比如长度） + 实际数据**

---

## 七、字符串在字节流里到底长什么样

现在我们可以抽象出它的结构。

如果把一个字符串写进 `DataStream`，它在流中的逻辑结构大致是：

```text
[STRING类型标识][长度信息][字符数据]
```

比如一个字符串 `"abc"`，逻辑上可以理解为：

```text
[STRING][3][a][b][c]
```

这里要注意两点：

### 1. 长度本身也不是裸写
长度是通过 `write(len)` 写入的。

而 `write(int32_t)` 本身又遵守统一规则，会先写 `INT32` 类型标识，再写 4 字节整数值。

所以严格来说，字符串内部的长度字段也带有自己的类型信息。

这意味着字符串的真实结构更接近：

```text
[STRING][INT32][长度值][字符内容]
```

这一点很重要，因为后面读取字符串时正是按这个结构来的。

---

### 2. 字符内容部分不再逐字符打类型标签
真正字符串内容那一段，是直接按字节连续写入的。

也就是说：
- 字符串整体有一个 `STRING` 外层标签
- 长度字段由 `INT32` 规则编码
- 具体字符内容直接作为原始连续字节写入

这是比较典型的“头部 + 负载”结构。

---

## 八、`read(string&)` 是怎么恢复字符串的

实现见：
- [../serialize/data_stream.cpp#L441-L456](../serialize/data_stream.cpp#L441-L456)

读取逻辑同样可以拆解为几个阶段。

### 第一步：检查当前位置是不是字符串类型
```cpp
if (m_buf[m_pos] != DataType::STRING)
{
    return false;
}
++m_pos;
```

这一步依旧延续统一规则：

- 先看当前位置的类型标识
- 如果不是 `STRING`，说明流结构和预期不匹配，直接失败
- 如果是，就让 `m_pos` 越过这个类型标识

这里你应该再次意识到：

> **类型标识的作用，不只是记录信息，更是读取阶段的校验机制。**

---

### 第二步：读取长度字段
```cpp
int len;
read(len);
```

这里很有意思。

它没有手动从字节里抠出长度，而是直接复用了整数读取逻辑。

也就是说：
- 字符串长度是一个标准的 `int32_t` 序列化值
- 读取长度时，直接调用 `read(int32_t&)`

这说明整个 `DataStream` 的实现思路是：

> **复用已有基础类型规则来搭更复杂的数据结构。**

这也是一个非常重要的设计习惯。

---

### 第三步：检查长度是否合法
```cpp
if (len < 0)
{
    return false;
}
```

这一步虽然简单，但很必要。

因为长度如果是负数，说明：
- 数据流可能已经损坏
- 或者当前读取位置出现错位
- 或者前面类型 / 内容解析出了问题

所以这里做了一个基本合法性判断。

---

### 第四步：按长度恢复字符串内容
```cpp
value.assign((char *)&(m_buf[m_pos]), len);
m_pos += len;
```

这一步的含义就是：

- 从当前位置开始取 `len` 个字节
- 用这 `len` 个字节构造 `std::string`
- 然后把 `m_pos` 向后推进 `len`

所以字符串恢复阶段的关键不是“找结束符”，而是：

> **按长度切片。**

这正是序列化格式处理中最稳定的方式之一。

---

## 九、为什么字符串序列化依赖长度而不是结束符

这个问题很值得专门想一遍。

如果不用长度，能不能只靠 `\0` 结束符？

理论上某些场景可以，但在通用序列化设计里，这样做通常不够稳健，原因包括：

### 1. 数据流是连续的
一个字段后面可能紧接着下一个字段。

如果没有显式长度，读取方就得靠扫描去找终点，这会让解析规则更脆弱。

---

### 2. 显式长度更高效
已知长度后，可以直接：
- 分配目标空间
- 一次性拷贝
- 精确推进读取位置

而不是逐字节查找边界。

---

### 3. 显式长度更通用
很多二进制数据、容器数据、嵌套结构，本来就不适合靠特殊终止符判断结束。

所以统一使用“长度 + 内容”的方式，会让整体设计更一致。

---

### 4. 长度控制更适合递归结构
后面你看 `vector`、`list`、`map` 时会发现，它们的边界管理也都是类似思想。

因此字符串这一课其实是在帮你建立：

> **变长数据必须有边界描述。**

---

## 十、字符串序列化和基本类型序列化的本质区别

你现在可以把两者做个对比。

### 基本类型
例如 `int32_t`：
- 类型固定
- 长度固定
- 读的时候只要知道类型，就知道该取多少字节

所以模式可以简化为：

```text
[类型][固定字节数据]
```

---

### 字符串
字符串：
- 类型固定
- 内容长度不固定
- 读的时候除了知道类型，还必须知道内容长度

所以模式变成：

```text
[类型][长度][内容]
```

这就是固定长数据和变长数据在序列化策略上的本质区别。

---

## 十一、你要开始注意 `DataStream` 的“分层搭建”思路

读到这里，你应该开始观察到一个很重要的设计特征：

> **复杂数据结构不是凭空实现的，而是建立在更基础的序列化规则之上。**

在字符串场景里表现得非常明显：

- 外层先写 `STRING`
- 长度字段复用 `write(int32_t)`
- 内容部分再直接按字节写入

这说明：

1. 类型标识机制是底层基础
2. 基本整数写入规则是底层基础
3. 字符串是在这些基础之上拼出来的复合结构

这会帮助你后面继续理解：
- 容器是怎么建立在字符串和基本类型之上的
- 自定义对象又是怎么建立在容器和基本类型之上的

---

## 十二、从字符串读写里可以学到哪些设计意识

这一课不只是看懂代码，更重要的是提炼设计意识。

### 1. 变长数据一定要有边界信息
这是最重要的一条。

---

### 2. 边界信息最好显式编码
不要依赖隐式约定，更不要依赖“碰巧能读出来”。

---

### 3. 复杂结构应尽量复用已有基础规则
字符串长度复用整数读写规则，就是一个典型例子。

---

### 4. 读取时必须先校验再推进
先检查 `STRING` 类型标识，再开始恢复内容，这比“直接硬读”稳定得多。

---

### 5. 解析过程本质上是在维护读取游标
整个恢复过程里，`m_pos` 的推进非常关键：
- 跳过类型标识
- 跳过长度字段
- 跳过内容区域

如果游标错位，后面所有解析都会出问题。

---

## 十三、把字符串序列化放进更大的系统背景里看

为什么这一课在中间件学习里很重要？

因为字符串其实是很多更复杂消息结构的起点。

现实系统中，大量信息最终都要落成某种字符串或字节序列：

- 名称
- 路径
- 配置项
- 主题名
- 节点标识
- 元数据字段
- 文本日志内容

所以如果你把字符串序列化真正看懂了，你就已经理解了一个非常关键的边界处理问题：

> **变长内容如何在连续字节流里被安全地表示和恢复。**

这会直接影响你后面看容器、协议、消息结构时的理解质量。

---

## 十四、本课小结

这一课最核心的结论是：

> **字符串是变长数据，因此仅有类型标识不够，必须额外记录长度，才能在连续字节流中被准确恢复。**

你现在应该记住：

- `DataStream` 对字符串使用了统一的类型标识机制，见 [../serialize/data_stream.cpp#L272-L288](../serialize/data_stream.cpp#L272-L288)
- 字符串写入遵循 `STRING + 长度 + 内容` 的基本模式
- 长度字段本身复用了整数序列化规则
- 字符串读取时先校验类型，再读取长度，再按长度恢复内容，见 [../serialize/data_stream.cpp#L441-L456](../serialize/data_stream.cpp#L441-L456)
- 变长数据的关键不是“内容是什么”，而是“边界在哪里”
- 这种“类型 + 元信息 + 负载”的模式会在后续容器序列化里继续出现
